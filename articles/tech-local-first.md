# はじめに
通常、WebサイトはアクセスするたびにサーバからHTMLやJavaScriptを取得します。

しかしPWAでは、一度取得したファイルを端末側へ保存し、その後はローカル環境からアプリを起動できます。

ここで疑問が生まれます。

「もし端末側にアプリ本体が存在するなら、サーバ上のファイルは本当に必要なのか？」

この記事では、PWAの仕組みからWebアプリの保存場所について考えます。

# 1. 通常のWebサイトとPWAの違い
通常：
```txt:
ブラウザ
 ↓
サーバ
 ↓
HTML/CSS/JS取得
 ↓
表示
```
毎回サーバが基準になります。

一方PWAでは：
```
初回

GitHub Pages
 ↓
Service Worker登録
 ↓
Cache Storage保存


2回目以降

ブラウザ
 ↓
Service Worker
 ↓
Cache Storage
 ↓
表示
```
になります。

つまりURLは同じでも、実際に動いているファイルは端末内に存在する可能性があります。

# 2. GitHubはアプリ本体ではなく配布場所になる
ここで役割が変わります。

従来：
```
GitHub
＝
ソースコード保存場所
```
でした。

しかしPWAでは、
```
GitHub
＝
初回配布場所

Cache Storage
＝
端末内アプリ本体
```
という関係になります。

一度インストールされた端末では、GitHub側のファイルがなくても動作する場合があります。

# 3. 「ファイルをリポジトリに置かない」という考え方
ここからが本題です。

常に更新する必要がないアプリなら、
```
端末
 ↓
Cache Storage
 ↓
自分だけ利用
```
という運用も可能です。

例えば、

[個人用メモアプリ，作業ツール，実験用アプリ]

などです。

GitHubは公開したい時だけ利用する。

つまり、
```
普段

端末内
 ↓
自分だけ利用


共有時

端末
 ↓
GitHub公開
 ↓
他人利用
```
という流れです。

# 4. ただしCache Storageは完全なファイル保存場所ではない
ここは注意点。

Cache Storageは、
```
URL
 ↓
レスポンス
```
を保存する仕組みです。

そのため、

アプリ本体：
```
HTML
CSS
JavaScript
画像
```
には向いています。

しかし、

ユーザーデータ：
```
メモ内容
設定
履歴
```
は別管理が適しています。

一般的には、
```
Cache Storage
↓
アプリ本体


IndexedDB
↓
データ
```
という分離になります。

# 5. これは未来のWebアプリ設計なのか
現在のWebは、

**サーバ中心**

から

**端末 + 同期**

へ変化しています。

現在：
```
サーバ
 ↓
端末
```

将来：
```
端末
 ↓
必要時だけ同期
 ↓
サーバ
```
という方向です。

これはLocal-firstという考え方に近いです。

# まとめ
PWAによって、Webアプリは単なる「サーバ上のページ」ではなくなります。

初回はWebから取得し、その後は端末内で動作する。

つまり、
```
GitHub
＝公開・配布場所

Service Worker
＝実行制御

Cache Storage
＝アプリ本体

IndexedDB
＝ユーザーデータ
```
という分離が可能になります。

Web技術は、再び「自分の端末に存在するソフトウェア」という方向へ近づいているのかもしれません。

# 追記：PWAから見える「ファイル管理」の変化

PWA自体は新しい技術ではありません。

また、今回GitHub Pagesを利用しましたが、公開場所はGitHubに限定されるものではなく、Netlifyなどのホスティングサービスでも同じ考え方ができます。

今回興味深かったのは、「どこにファイルを置いておくべきなのか」という点でした。

通常のWebアプリでは、
```
サーバ
 ↓
ユーザー
```
という関係になります。

しかしPWAでは、
```
サーバ
 ↓
端末内キャッシュ
 ↓
ローカル動作
```
という形になります。

つまり、サーバ上のファイルは「常に参照する本体」ではなく、「端末へ渡すための入口」と考えることもできます。

もちろん、単純なアプリであればアプリストアから取得すれば済む話です。

しかし、自分だけが使う作業ツールや、外部サービスへデータを預けることに不安を感じる用途では、PWAによって「自分の端末内で完結するWebアプリ」を構築できることに大きな意味があります。

ここで疑問になるのは、将来的にファイルという概念はどこまで必要なのかという点です。

現在は、
```
ファイルを保存する
↓
必要時に読み込む
```
という考え方が一般的です。

しかし、ブラウザ内ストレージや同期技術が発展すると、
```
アプリ状態
↓
同期
↓
必要な場所で復元
```
という管理方法が増える可能性があります。

つまり「ファイルを移動する」のではなく、「状態を同期する」という考え方です。

ただし、Cache Storageは万能な保存領域ではなく、容量や削除条件などブラウザ側の管理に依存します。

そのため現状では、
```
Cache Storage
→ アプリ本体

IndexedDB
→ ユーザーデータ

サーバ
→ 共有・バックアップ
```
のような役割分担が現実的です。

PWAを触っていると、Webアプリは単なる「サーバ上のページ」ではなく、「端末上に展開されるソフトウェア」へ近づいているように感じます。
